### F1 · [MEDIUM] `make check-shell` can never fail — the shell-syntax CI gate is decorative

- **Location:** `Makefile:23`  ·  **dimension:** build-packaging-runtime  ·  **claimed severity:** MEDIUM

- **Evidence.** Line 22-23: `check-shell: ## Validate shell script syntax (hooks, installer, probes)` / `@for f in .githooks/pre-commit hooks/*.sh probe/*.sh; do [ -e "$$f" ] && sh -n "$$f" \|\| true; done; echo "Shell scripts syntax OK."`

- **Why it matters.** The `\|\| true` swallows every non-zero `sh -n` exit, and the recipe's last command is a successful `echo`, so the recipe ALWAYS returns 0. I demonstrated this: a script with an unterminated `if [` makes `sh -n` return 2, but the loop line exits 0. Consequence: `make ci` (the only thing GitHub Actions runs, .github/workflows/check.yml:18) and the pre-commit hook (.githooks/pre-commit:15) can never reject a syntax-broken shell script. This is not cosmetic: guard-destructive.sh is a *blocking* PreToolUse hook (opt-in) — dash returns exit code 2 on a parse error, which is exactly the Kimi 'block' signal (references/kimi-runtime.md:60), so a syntax typo shipped in that hook would BLOCK every Bash call in every Kimi session, and the one gate meant to catch it is a no-op. Compounding it, the glob omits `scripts/install.sh` and `scripts/install-hooks.sh` entirely, so the two installer scripts the help text claims to cover ('installer') are never even fed to `sh -n`.

- **Suggested direction.** Make the recipe fail on first bad parse: drop `\|\| true`, iterate with a failure flag (`for f in ...; do sh -n "$f" \|\| rc=1; done; exit $rc`) or `set -e` the loop; and extend the glob to include `scripts/*.sh` so the installers are actually checked. Prefer `bash -n` for the bash-targeted files if any use bashisms.

- **Adversarial verdict.** CONFIRMED — Reproduced the exact recipe (Makefile:23) in a mirror dir with a syntax-broken script (unterminated `if [`): `sh -n` returned exit 2 and printed a syntax error, but the recipe still printed "Shell scripts syntax OK." and `make` exited 0. The `[ -e "$f" ] && sh -n "$f" \|\| true` per-file `\|\| true` swallows every non-zero parse result, and the loop's trailing `echo` (always exit 0) is the recipe's final command, so `make check-shell` structurally cannot fail. This is the only shell-syntax gate reached by `make ci` (the sole thing .github/workflows/check.yml:18 runs) and by the opt-in .githooks/pre-commit:15 (which also calls `make ci`), so neither can reject a syntax-broken shell script. Coverage-gap claim also confirmed: scripts/install.sh and scripts/install-hooks.sh exist but the glob `.githooks/pre-commit hooks/*.sh probe/*.sh` never feeds them to `sh -n`, contradicting the help text's "installer" claim (Makefile:22). The guard-destructive.sh blast-radius argument is real (it is the PreToolUse blocking hook; kimi-runtime.md:60 confirms exit 2 = block) but partially mitigated because PLAN.md:16/133/332 document it ships DISABLED by default. Not an intentional documented tradeoff — no rationale exists for a gate that always passes; it is simply a bug. MEDIUM is accurate.


### F2 · [LOW] `guard-destructive.sh` denylist is trivially bypassable and the header over-claims

- **Location:** `hooks/guard-destructive.sh:95`  ·  **dimension:** security  ·  **claimed severity:** MEDIUM → **corrected:** LOW

- **Evidence.** Target regex requires whitespace immediately before the root token: match '([[:space:]](/\|/\*\|~\|\$HOME\|\$\{HOME\})([[:space:]]\|/\|\*\|$)\|--no-preserve-root)'; and command-position wrapper CMDPOS='(^\|[;&\|<>(){}`])[[:space:]]*((sudo\|env\|command\|exec\|nohup)[[:space:]]+)*' (line 81) only whitelists sudo/env/command/exec/nohup. Empirically tested: command `rm -rf "/"` -> exit 0 (ALLOW); `FOO=bar rm -rf /` -> exit 0 (ALLOW); `rm -rf '/'*` -> exit 0 (ALLOW); while bare `rm -rf /` -> exit 2 (DENY).

- **Why it matters.** The header (lines 90-96) states purpose #2 is 'Recursive+forced rm of the root ... so rm -rf / is not [allowed]'. But quoting the target (rm -rf "/") breaks the [[:space:]]/ adjacency and is ALLOWED; prefixing a benign env-assignment (FOO=bar rm -rf /) moves rm off command-position (CMDPOS omits VAR=val prefixes) and is ALLOWED. An agent, or injected content the model relays into Bash, wipes the root exactly as the guard promises to prevent. It is opt-in/disabled by default, which caps blast radius, but as written it does not deliver its own named guarantee once enabled. DOCUMENTED-TRADEOFF: the narrow denylist is an intentional design choice, but the rationale explicitly claims to catch rm -rf / and that specific claim is demonstrably false under one layer of quoting.

- **Suggested direction.** Denylists cannot be made complete; if this is meant to be a real control, normalize the command (strip quotes, resolve leading VAR=/wrapper tokens) before matching and add VAR=val to the CMDPOS alternation. Otherwise weaken the header claim from 'rm -rf / is denied' to 'best-effort, trivially bypassable' so operators do not over-trust it before the P4b probe wires it in.

- **Adversarial verdict.** CONFIRMED — Opened hooks/guard-destructive.sh and reproduced all four claimed cases by feeding real PreToolUse JSON to the script: `rm -rf /` -> exit 2 (DENY); `rm -rf "/"` -> exit 0 (ALLOW); `FOO=bar rm -rf /` -> exit 0 (ALLOW); `rm -rf '/'*` -> exit 0 (ALLOW). The mechanism holds as claimed: the target regex at line 95 requires literal `[[:space:]]` immediately before the root token, so a quote char (`"/"` or `'/'`) between the space and `/` defeats every alternation branch; and CMDPOS at line 81 whitelists only sudo/env/command/exec/nohup, so a leading `VAR=val` assignment moves `rm` off command position and the line-92 rm check fails. The header at line 91 and the deny reason at line 96 both explicitly promise that `rm -rf /` is denied — that specific guarantee is demonstrably false under one layer of quoting. This is not merely the documented 'narrow denylist' tradeoff: the narrowness rationale does not license asserting coverage of `rm -rf /` while a single quote bypasses it. Severity lowered from MEDIUM to LOW because the hook is DISABLED BY DEFAULT, not wired into plugin.json, gated behind the unfinished P4b probe (lines 4-15), and is defense-in-depth behind the permission system / human gates, so there is no live exploit path today; the actionable defect is the over-confident header claim, which should be softened to 'best-effort, trivially bypassable'.


### F3 · [LOW] SAST floor uses `semgrep --config auto` — silent third-party egress / metrics not disabled

- **Location:** `scripts/sast.py:185`  ·  **dimension:** security  ·  **claimed severity:** MEDIUM → **corrected:** LOW

- **Evidence.** argv = [executable, "--config", "auto", "--json", "--quiet", "--", *paths]. `--config auto` reaches out to semgrep.dev to fetch rules and, per semgrep's documented default, sends usage metrics when Registry/auto rules are used. There is no --metrics=off, no --offline, and no pinned local ruleset. The docstring (lines 12-16) documents FAIL-OPEN on rule-fetch failure but says nothing about egress.

- **Why it matters.** Every SECURITY-lens run over a private diff triggers an outbound connection to a third party (rule fetch plus default metrics), surprising for a tool auditing confidential code in a regulated/air-gapped context. Separately, because the floor is mandatory-fail-open, any CI/runner without egress to semgrep.dev makes the 'deterministic SAST floor' silently contribute zero findings while the rubric/README present it as a real backstop — a security control quietly absent exactly where the network is locked down. DOCUMENTED-TRADEOFF: fail-open is a deliberate design decision, but the network egress/telemetry side effect is neither fail-open-related nor documented, and disabling metrics does not weaken fail-open.

- **Suggested direction.** Add --metrics=off (or SEMGREP_SEND_METRICS=off) to the argv and prefer a pinned/vendored local ruleset over --config auto so the floor is deterministic and egress-free; if remote fetch is kept, document the egress so operators can make an informed choice.

- **Adversarial verdict.** CONFIRMED — The claim is compound; I split it.  PRONG A — "offline zero coverage / silently absent backstop" is REFUTED as a flaw. The degradation is exhaustively and honestly documented as an intentional design decision, not a hidden gap: sast.py:11-16 ("FAIL-OPEN is mandatory... the network rule-fetch (--config auto) fails... returns no findings and the SECURITY lens degrades to exactly today's judgment-only behavior"); SKILL.md:357-360 ("fail-open and OPTIONAL: if... its --config auto rule-fetch fails, sast.scan returns [] and the SECURITY lens degrades to exactly today's judgment-only behavior... SAST augments it, it does not replace it"); rubric.md:84-95 (labels it "PARTIAL", "OPTIONAL and fail-open", degrades "if... the --config auto rule-fetch fails"); README.md:134 explicitly calls it "fail-open semgrep sast". Nothing presents it as a mandatory backstop, so "a security control quietly absent" overstates a documented, sound tradeoff. The rule-fetch network dependency itself is also disclosed in all three docs, so the egress is not "silent."  PRONG B — undocumented default telemetry egress is CONFIRMED and actionable. Line 185's argv omits --metrics=off and line 187 passes no env=, so semgrep's default (verified from the installed 1.169.0 help: "--metrics absent defaults to auto", metrics "sent whenever the --config value pulls from the Semgrep server") means every SECURITY-lens scan of a private diff sends pseudonymous usage metrics to semgrep.dev. This is real, reachable (semgrep is installed at /root/.local/bin/semgrep so scan() actually runs), avoidable, and — unlike the rule-fetch reliability aspect — the privacy/telemetry dimension is documented nowhere. The suggested fix (--metrics=off / SEMGREP_SEND_METRICS=off) is trivial and does not weaken fail-open or determinism.  Severity corrected MEDIUM→LOW: the telemetry is pseudonymous scan metadata, NOT source-code content (the claim's "scanned-code metadata" phrasing overstates this), the rule-fetch egress is already disclosed, and the offline-coverage half of the claim is a documented intentional tradeoff. What remains is a genuine but minor privacy/hygiene gap: an undocumented, default-on third-party telemetry beacon on every scan of confidential code, fixable with one flag.

### F6 · [LOW] Rubric vocabulary (`_DIMENSIONS`/`_BLOCKING`) copy-pasted across pure cores

- **Location:** `scripts/verdict.py:29`  ·  **dimension:** architecture-design  ·  **claimed severity:** MEDIUM → **corrected:** LOW

- **Evidence.** verdict.py:29-36 declares `_DIMENSIONS: tuple[str, ...] = ("CORRECTNESS", "CODE-QUALITY", "SECURITY", "TEST-ADEQUACY", "DOES-IT-RUN", "REQUIREMENTS-COVERAGE")` — byte-identical to quality.py:26-33. `_BLOCKING = {"CRITICAL", "HIGH"}` is declared three times: verdict.py:24, quality.py:35, run_negative_gate.py:101. Frontmatter parsing is not among the documented intentional-design decisions, so this is an ordinary latent inconsistency, distinct from the sanctioned path-shim boilerplate.

- **Adversarial verdict.** CONFIRMED — Verified all cited facts. skillregistry.py:73 regex is CRLF-aware (`\r?\n`) but not BOM-aware and parse_frontmatter (line 111) raises ValueError on no match; run_negative_gate.py:131 regex is BOM-aware (optional literal U+FEFF) but LF-only (`\n`), and strip_frontmatter (220-231) silently returns text unchanged on no match — genuinely opposite blind spots. skillextract.py:162 does reuse skillregistry.parse_frontmatter, so exactly two implementations diverge with no shared canonical helper. Minor overstatement in the title ('three' parsers) is self-corrected in the evidence body.


### F8 · [LOW] `reqcoverage` doesn't strip trailing tab+timestamp from `+++` diff headers

- **Location:** `scripts/reqcoverage.py:29`  ·  **dimension:** correctness  ·  **claimed severity:** LOW

- **Evidence.** `_NEW_PATH_RE = re.compile(r"^\+\+\+ (?:b/)?(.+)$", re.MULTILINE)` (reqcoverage.py:29) captures `.+` greedily to end-of-line, and _changed_paths (reqcoverage.py:77-84) only does `p.strip()` — it never splits on the tab. A `+++ b/foo.py\t2026-01-01 ...` header (unified diffs may carry a tab + timestamp after the path) yields the path `foo.py\t2026-01-01 ...`, which then fails the `_under_scope` prefix comparison. By contrast integrate.touched_files.add (integrate.py:36) explicitly does `raw.split("\t", 1)[0]` to drop this metadata.

- **Why it matters.** A changed file whose diff header carries the optional trailing tab+timestamp is compared against scope_paths with the timestamp still attached, so `_under_scope` returns False and the file is wrongly flagged as MEDIUM scope-creep (a false red that wastes refine attention). Advisory-only (MEDIUM, can never gate), and git's own `git diff` output normally omits the trailing tab, so exposure is limited — hence LOW.

- **Suggested direction.** Mirror integrate.touched_files: split the captured path on the first tab (`p.split('\t',1)[0]`) before stripping, so both diff parsers canonicalize paths identically.

- **Adversarial verdict.** CONFIRMED — Independently reproduced. reqcoverage.py:29 `_NEW_PATH_RE = r"^\+\+\+ (?:b/)?(.+)$"` captures `.+` to end-of-line, and `_changed_paths` (reqcoverage.py:81) only does `p.strip()`, which removes surrounding whitespace but NOT an interior tab. Feeding a POSIX-style unified diff header `+++ b/foo.py\t2026-01-01 ...` to `coverage([], diff, ['foo.py'])` produced a false SC0 MEDIUM scope-creep defect with location `foo.py\t2026-01-01 ...` because `_under_scope` (reqcoverage.py:87-93) does an exact/prefix compare against the timestamp-laden path and returns False. ### F10 · [INFO] Negative-gate tests leak alarming uncaptured stdout into the passing suite

### F11 · [LOW] `install.sh` accumulates unbounded `installed.json.bak.<ts>` backups

- **Location:** `scripts/install.sh:65`  ·  **dimension:** build-packaging-runtime  ·  **claimed severity:** LOW

- **Evidence.** Line 65 (install) and line 34 (uninstall): `[ -f "$INSTALLED" ] && cp "$INSTALLED" "$INSTALLED.bak.$(date -u +%Y%m%dT%H%M%SZ)"`.

- **Why it matters.** Every install AND every uninstall drops a new `installed.json.bak.<UTC-second>` next to installed.json in the user's `~/.kimi-code/plugins/` dir, with no rotation or cleanup — a re-runnable installer (explicitly advertised as idempotent, line 5) therefore litters the plugins directory without bound over repeated dev cycles. Secondly, the timestamp granularity is one second, so two installs within the same second overwrite the same backup filename, silently losing the earlier snapshot the backup was meant to preserve.

- **Suggested direction.** Keep a single rolling backup (`installed.json.bak`) or prune to the last N; if timestamped backups are desired, add a monotonic suffix or nanoseconds to avoid same-second collisions.

- **Adversarial verdict.** CONFIRMED — Verified lines 65 (install) and 34 (uninstall) verbatim: `[ -f "$INSTALLED" ] && cp "$INSTALLED" "$INSTALLED.bak.$(date -u +%Y%m%dT%H%M%SZ)"`. There is no rotation/pruning/cleanup of `installed.json.bak.*` anywhere in the script, and the header (lines 5-7) explicitly advertises the installer as idempotent/re-runnable, so every install AND every uninstall unconditionally litters `$KIMI_HOME/plugins/` with another backup — the unbounded-accumulation half of the claim genuinely holds and cannot be refuted. It is not among the documented intentional tradeoffs (the header only promises "backed up and written atomically," never rotation) nor a known D1-D7 item. The same-second self-collision half is real in code but practically unreachable on the install path: line 65 runs only after `git archive HEAD \| tar -x` of the whole repo (lines 59-61), which extracts the deliberately-kept 73MB skills/xlsx binary, so two full installs cannot finish within one UTC second; it is only marginally reachable via two back-to-back `--uninstall` runs (line 34 has no archive step), where the second overwrites the first backup with already-cleaned state — negligible data. Net: a genuine but purely cosmetic operational wart. Impact is directory clutter of tiny JSON snapshots, no functional/resource harm, so LOW (claimed LOW) is appropriate; not intentional_tradeoff.
